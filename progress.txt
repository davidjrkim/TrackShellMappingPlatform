## Codebase Patterns
- Review UI mutation pattern: each correction control (reassign hole, change type, etc.) lives in its own sub-component inside Inspector.tsx with its own local `selectedValue`, `submitting`, and `error` state. `useEffect` resets local state on `feature.id` change so switching polygons doesn't carry stale selections. Apply button: `disabled = submitting || !dirty`. No optimistic updates — keep the original feature data visible until the parent refresh lands the new server state. Surface non-2xx as inline `role="alert"` text, not a global toast.
- Inspector consumes `holes: HoleSummary[]` (full list for the course) so the reassign dropdown can render `Hole 1..N` without an extra fetch — the holes prop is already in ReviewWorkspace.
- Map GeoJSON refresh after a successful correction goes through a memoised `refreshGeoJSON()` in ReviewWorkspace (used on mount AND after mutations). For hole-detail features, bump a `holeFeaturesVersion` state in ReviewWorkspace and include it in the hole-detail useEffect deps array — that forces a refetch without coupling to the GeoJSON load.
- After PATCH /hole succeeds, the polygon may belong to a different hole; switch `selectedHoleId` to the new hole id (server returns it) so Inspector + map stay focused on the moved feature, and clear `selectedFeatureId` since the new hole's feature list will land via the standard hole-detail effect.
- Feature type options for the UI: hardcode the array (`['green','fairway','tee_box','bunker','water_hazard'] as const`) in the client component — do NOT import `FeatureType` from `@prisma/client` into a 'use client' file because it pulls the runtime client into the browser bundle. The list is already mirrored in FEATURE_LABEL/FEATURE_SWATCH so keep them in sync if a new type is added.
- Correction endpoints share a preflight flow: authenticate → UUID-validate featureId → parse+validate body → getFeatureForMutation (org isolation) → isLockedByOther (409) → body-specific 422s → $transaction([INSERT corrections, UPDATE/DELETE features]).
- Every correction mutation MUST run as a Prisma $transaction with the corrections INSERT first and the feature mutation second, so a half-applied correction cannot ship.
- corrections.original_geometry and features.geometry are both GEOMETRY(MultiPolygon,4326). Snapshot the prior geometry with INSERT ... SELECT f.geometry FROM features (no ST_AsGeoJSON round-trip needed).
- Enums need explicit ::feature_type_enum casts in raw SQL when passed as string bind parameters.
- lib/spatial.ts isGeometryValid(geojson) returns { valid, areaSqm } from ST_IsValid + ST_Area(::geography) in one query — reuse it rather than writing new raw SQL in route handlers.
- Use ST_Multi(ST_GeomFromGeoJSON(...)) when writing to a MultiPolygon column to tolerate single-Polygon GeoJSON without failing the type constraint.
- lib/corrections.ts exposes getFeatureForMutation / getHoleRef / isLockedByOther — always use these, do not re-query in routes.
- UUID regex used for all featureId / holeId params: /^[0-9a-f]{8}-[0-9a-f]{4}-[0-9a-f]{4}-[0-9a-f]{4}-[0-9a-f]{12}$/i.
- corrections.feature_id has ON DELETE SET NULL — the audit row survives a hard DELETE of the feature, so correction snapshots are the only record after polygon_delete.
- corrections has no column for feature confidence_score or a raw hole_id — snapshot confidence_score into the notes TEXT column as JSON, and hole identity into original_hole_number via LEFT JOIN holes h ON h.id = f.hole_id.
- Hole-scoped correction events (e.g. hole_confirmed) set corrections.feature_id = NULL and record hole identity via original_hole_number. feature_id is nullable by design.
- holes.confirmation_type is a nullable hole_confirmation_type_enum ('manual' | 'auto') — set by US-005 (manual) and US-006 (auto); cast string literals explicitly in raw SQL: `'manual'::hole_confirmation_type_enum`.
- Course-scoped transactions (e.g. sign-off in US-006) should release the lock inside the same $transaction that flips status — do NOT call releaseLock() separately after the transaction, or a partial failure can leak the lock. UPDATE courses SET status=..., locked_by=NULL, locked_at=NULL in one statement.
- Sign-off gate query is `SELECT hole_number FROM holes WHERE course_id=$1 AND needs_review=true AND confirmed=false` — return the list as `blocking_holes: number[]` in the 400 body so the UI can render which holes still block sign-off.
- Prisma + PostGIS interaction: `prisma migrate dev` refuses to run non-interactively and complains about the `bounding_box` PostGIS column it cannot see. Workaround when adding a plain column: create the migration folder under prisma/migrations/<timestamp>_<name>/migration.sql by hand with the ALTER/CREATE TYPE, then `npx prisma migrate deploy` to apply it and record it in _prisma_migrations. Update prisma/schema.prisma alongside so the generated client picks up the new column.
- Read endpoints that scope by course_id must still org-isolate: do a preflight `SELECT id FROM courses WHERE id=$1 AND org_id=$2 AND deleted_at IS NULL` (404 if empty), then issue the data query. corrections has no org_id column — the course join / preflight is the only barrier.
- For corrections audit reads, expose geometry snapshots as `{ has_geometry_snapshot: boolean, snapshot_area_sqm: float|null }` via `ST_Area(original_geometry::geography)`. Do NOT ship full MultiPolygon blobs in list payloads — they are large, and the UI only needs presence + size for the audit feed.
- @mapbox/mapbox-gl-draw works with MapLibre GL v4 if you tag the canvas with the `mapboxgl-canvas` class while Draw is mounted — Draw's pointer-event guard (`src/events.js#132`) checks `classList.contains('mapboxgl-canvas')` and MapLibre's canvas is named `maplibregl-canvas`. Add the class on mount, remove on unmount so we don't leak the Mapbox class after leaving draw mode.
- @mapbox/mapbox-gl-draw ships no types; keep a minimal ambient module in `types/mapbox-gl-draw.d.ts` covering only the surface we touch (constructor, add, changeMode, DrawUpdateEvent). Don't try to fully re-type the library.
- When mounting a Draw control onto a MapLibre map, `map.addControl(draw as unknown as maplibregl.IControl)` is the correct cast — Draw's control shape is compatible but the types don't overlap.
- To get the MapLibre map instance from MapCanvas for sibling components (DrawMode overlay etc.), add a `getMap(): MapLibreMap | null` method to MapCanvasHandle AND expose an `onMapReady(map)` prop that fires from the map `load` event — state-backed is needed because a ref doesn't trigger re-renders.
- Review UI keyboard handlers: when draw mode is active, short-circuit the default Escape handler that deselects the feature — otherwise a single Esc both exits draw mode and deselects. Add `drawModeActive` as a dep to the keydown effect and guard the Escape branch with `if (drawModeActive) break`.
- Draw's direct_select mode works per-polygon. When seeding a MultiPolygon, unwrap/rewrap via helpers: accept either Polygon or MultiPolygon back from draw.update and normalise to MultiPolygon before POSTing (PATCH /geometry requires MultiPolygon and lib/spatial.ts ST_Multi wraps any residual Polygon anyway).
- Destructive modals (e.g. DeletePolygonDialog) should attach their Escape-to-cancel listener with useCapture=true AND stopPropagation(), because the ReviewWorkspace has its own window-level Escape handler that deselects the feature — without the capture+stop combo a single Esc both closes the dialog AND clears selection. Back it up with a `if (deleteDialogFeature) break` guard in the workspace handler for safety.
- Keyboard shortcuts that mutate state (Delete/Backspace to open delete dialog, D to toggle draw mode) must be listed in the keydown effect's dep array — not just the state they read. Otherwise the effect closes over stale booleans like drawModeActive/deleteDialogFeature and the guards silently fail.
- Render overlay modals at the ReviewWorkspace level, not inside a side panel — the right-panel Inspector has its own stacking context, which breaks a `fixed inset-0` overlay's positioning. The workspace root is the correct anchor for any full-screen dialog.
- Single-level undo pattern: ReviewWorkspace owns a `LastCorrection` discriminated union that records the INVERSE payload after each successful correction. Success callbacks from child controls carry the prior value (Inspector's ReassignHoleControl/ChangeTypeControl now pass {featureId, priorHoleId|priorType}); geometry prior is captured from the existing `selectedFeatureGeometry` memo before the refresh lands. Undo dispatches a fetch DIRECTLY from the workspace rather than re-entering the Inspector flow — re-entering would re-fire the success callback and register an undo-of-undo, defeating single-level. On success clear the slot + refresh; on failure surface inline error and leave the slot so the user can retry.
- For non-reversible corrections (polygon_delete — no feature-insert endpoint yet), record a `{undoable: false, reason}` slot instead of leaving the slot null. That lets the Undo button render disabled WITH a tooltip explaining why, which satisfies the AC's "otherwise undo is disabled with tooltip" clause. A null slot would just hide the control silently.
- Global Enter keyboard shortcut and focused buttons double-fire: when Enter is pressed with a BUTTON or A focused, the browser already dispatches a synthetic click on that element. A window-level `keydown` that also fires on Enter will double-action. Guard the Enter branch with `if (e.target.tagName === 'BUTTON' || tag === 'A') break` so the native click is the only thing that happens when something clickable is focused. `isTypingTarget` handles INPUT/TEXTAREA/SELECT/contentEditable but deliberately does NOT exclude BUTTON — other shortcuts (↑↓, F, C) should still work when a button has focus.
- Per-hole confirm from the client: POST `/api/courses/{id}/holes/{holeId}/confirm` writes the corrections row + flips holes.confirmed server-side. The client must mirror the flip in its local `holes` state so HoleList re-sorts and re-colours without a page reload — change `const [holes] = useState(initialHoles)` to include a setter, then `setHoles(prev => prev.map(h => h.id === id ? {...h, confirmed:true, needs_review:false} : h))` on 2xx. Also bump `holeFeaturesVersion` + call `refreshGeoJSON()` so features.reviewed=true lands in Inspector/overlay.
- Sign-off UI reachability: compute `hasBlockingFlagged = holes.some(h => h.needs_review && !h.confirmed)` on the client as the gate for the "Sign off course" CTA. The server will re-check the same condition on POST /review/complete and return 400 with `blocking_holes: number[]`, so the client gate is UX polish, not a security boundary — always render the server error path too.
- When a workspace-level modal (DeletePolygonDialog, SignOffDialog) is open, short-circuit the window `keydown` handler at the top (`if (signOffOpen) return`) in addition to whatever per-branch guards exist. Otherwise typing in the notes textarea is fine (isTypingTarget catches that), but clicking outside the textarea still leaves every other shortcut (D, Enter, Ctrl+Z, fit-to-hole) hot on the map underneath. One top-level guard per modal is cleaner than sprinkling guards in every case.
- For dialogs that navigate away on success, call both `router.push(path)` AND `router.refresh()` — push alone uses the cached server component render (the course status still reads as 'flagged' until the next hard reload). Refresh invalidates the RSC cache so the new 'reviewed' status lands on first paint of the overview.
- Auto-advance-on-confirm pattern: after the confirm-hole 2xx lands, compute the next flagged hole from a LOCAL `updated` array (not from the `holes` state, which won't have the new confirmed=true yet) and setSelectedHoleId to it. The existing re-center useEffect on selectedHoleId handles fitToHole, so no explicit mapRef.fitToHole call is needed. Sort order: assignment_confidence ASC, then hole_number ASC as tiebreaker — matches HoleList.orderHolesForReview so the UI focus matches what the reviewer sees at the top of the list.
- To compute derived next-selection state from a just-updated list while avoiding nested setters, refactor `setHoles(prev => prev.map(...))` to `const updated = holes.map(...); setHoles(updated); if (nextFrom(updated)) setSelectedHoleId(...)`. React is fine with two independent setStates in the same callback; batching handles the re-render. Keep the `holes` dep in useCallback so the mapping reads the latest snapshot.
- Transaction-integrity test pattern (US-016): to prove a route handler's `db.$transaction([a, b])` is atomic without mocking Prisma entirely, capture `const realTransaction = db.$transaction.bind(db)` at module scope, then per-test `jest.spyOn(db, '$transaction').mockImplementationOnce(arg => realTransaction([arg[0], db.$executeRaw\`SELECT 1/0\`]))`. The real $transaction still runs the modified array in one Postgres tx, so the forced division-by-zero rolls back the sibling statement. Verifies atomicity from outside the route — no need to reach into Prisma internals. Reset mocks with `jest.restoreAllMocks()` in afterEach so one test's stub doesn't bleed into the next.
- Seeding a feature row for tests requires raw SQL because Prisma cannot write PostGIS geometry: `INSERT INTO features (id, hole_id, course_id, feature_type, geometry) VALUES (gen_random_uuid(), $hole::uuid, $course::uuid, 'green'::feature_type_enum, ST_Multi(ST_GeomFromText('POLYGON((...))', 4326))) RETURNING id`. Use a polygon larger than 20 m² (the floor PATCH /geometry enforces) — e.g. 0.001° × 0.001° at latitude 37° ≈ 9,900 m².

- Implemented PATCH /api/features/[featureId]/geometry to save edited polygon geometry with an atomic corrections audit.
- Files changed:
  - app/api/features/[featureId]/geometry/route.ts (new)

- Flow:

  - `lib/spatial.ts` already has `isGeometryValid(geojson)` doing exactly the ST_IsValid + ST_Area(::geography) pair required by the PRD — do not duplicate.
  - Prisma `$transaction` with an array of `$executeRaw` statements guarantees atomic rollback across raw SQL — no need for interactive transaction callback for two-statement correction flows.
  - Area threshold is 20 m² (below which we reject as likely editing noise / drag slips).

## 2026-04-24 - US-006
- Implemented POST /api/courses/[id]/review/complete — course sign-off with atomic auto-confirm, status flip, and lock release.
- Files changed:
  - app/api/courses/[id]/review/complete/route.ts (new)
  - prd.json (US-006 passes=true + notes)
  - progress.txt (this entry + Codebase Patterns update)
- Flow:
  1. Session required (401) + UUID-validate params.id (404).
  2. Parse optional body `{ notes?: string }` — tolerate no body, empty string trims to null.
  3. getLock(courseId, orgId) — org-isolated; null → 404.
  4. isLockedByOther(lock.locked_by, lock.locked_at, userId) → 409.
  5. Sign-off gate: SELECT hole_number FROM holes WHERE course_id=$1 AND needs_review=true AND confirmed=false — if any rows, return 400 with `blocking_holes: number[]`.

- **Learnings for future iterations:**
  - Lock release for course-scoped writes belongs INSIDE the same transaction as the state flip. If you call releaseLock() after a $transaction completes, a crash between the two leaves the lock dangling and the course un-reviewable until the 2h TTL expires. Inline `SET locked_by=NULL, locked_at=NULL` into the UPDATE courses SET status=... statement instead.

  - corrections.correction_type is a VARCHAR(50) (not an enum), so new event types like 'course_signoff' require no migration — just pick a stable string.
  - Auto-confirm path leaves confirmation_type='auto' on holes that never had needs_review flag set AND were never manually confirmed. The WHERE `confirmed=false` after the gate is safe because the gate guaranteed zero `needs_review=true AND confirmed=false` rows remain.
  - Body parsing pattern: `await req.json()` in a try/catch with null fallback — Next.js throws on empty body, so tolerate it for optional-body POSTs.
