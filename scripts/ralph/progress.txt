## Codebase Patterns
- Route handlers follow `getServerSession(authOptions)` → 401, UUID regex check → 404, org isolation via `session.user.orgId` joined to `courses.org_id`. See `app/api/courses/[id]/route.ts` for canonical shape.
- Raw SQL PostGIS helpers live in `lib/spatial.ts`; non-geometry mutation helpers (correction flow) live in `lib/corrections.ts`. Route handlers must not embed raw SQL directly — call a helper.
- Correction mutations MUST use `db.$transaction([...])` with `$executeRaw` statements so the corrections insert and the features/holes mutation roll back together. Correction row first, mutation second, per PRD 2b.
- The corrections table stores hole NUMBERS (`original_hole_number` / `corrected_hole_number` smallints), not hole ids. Resolve the hole_number from the hole row before writing the snapshot.
- Lock enforcement for mutation endpoints: read `courses.locked_by` + `locked_at` via `getFeatureForMutation` and gate with `isLockedByOther` (respects the 2h TTL from `LOCK_TTL_MS` in `lib/review.ts`). Do NOT call `acquireLock` — that belongs to the explicit POST /lock endpoint.
- Path params for nested dynamic routes arrive as `{ params: { featureId: string } }`. Validate with `UUID_RE` before touching the DB.
- Prisma client type `@prisma/client` exposes `FeatureType` enum string literals (`'green' | 'fairway' | 'tee_box' | 'bunker' | 'water_hazard'`) for type-change endpoints.
- Enum bindings in `$executeRaw` must be cast explicitly: `${value}::feature_type_enum`. The `@@map` in schema.prisma names the DB type `feature_type_enum` (not `feature_type`).
