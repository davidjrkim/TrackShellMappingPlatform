## 1. Purpose & Scope

This PRD defines the complete database schema for the TrackShell Course Mapping Platform. The database is the foundation layer that all other components — the ML pipeline, the web dashboard, the manual review UI, and the consumer API — read from and write to. Getting this right before any other build work prevents costly migrations later.

**Scope:**
- ✅ Covers: table definitions, column types, indexes, relationships, constraints, and geospatial configuration.
- ❌ Does not cover: ORM models, migration tooling, or query patterns (covered in respective feature PRDs).

---

## 2. Technology Decisions

### 2.1 PostgreSQL + PostGIS

Golf course polygon data is fundamentally geospatial. A standard relational database without geospatial extensions would require storing coordinates as raw floats and computing spatial queries in application code — slow, fragile, and hard to maintain. PostGIS gives us:

- `ST_Contains()` — find which hole a GPS point falls inside in a single query
- `ST_Distance()` — compute yardage between two points natively in the DB
- `ST_Simplify()` — optional server-side polygon simplification on read
- Native geometry column type for storing GeoJSON polygons efficiently
- Spatial indexes (GIST) — fast bounding-box queries across thousands of polygons

> **WHY NOT MongoDB:** MongoDB with GeoJSON support was considered but rejected. Geospatial query performance at scale is significantly better in PostGIS, and the relational structure (course → holes → features) maps cleanly to PostgreSQL.

### 2.2 Coordinate Reference System

All geometry is stored in **WGS84 (SRID 4326)** — the same CRS used by GPS, Google Maps, and Kakao Maps. This avoids projection conversion at query time and ensures compatibility with all downstream consumers.

**Enums:**

```sql
CREATE TYPE course_status AS ENUM (
  'unmapped',    -- Added to system, no processing started
  'processing',  -- ML pipeline currently running
  'segmented',   -- Polygons extracted, hole assignment pending
  'assigned',    -- Holes assigned by LLM, awaiting review
  'reviewed',    -- Human reviewed and confirmed
  'published',   -- Live on consumer API
  'failed'       -- Pipeline error, needs investigation
);

CREATE TYPE data_source_type AS ENUM (
  'ml_pipeline', -- Generated by DeepLabv3+ + LLM assignment
  'manual',      -- Manually entered by operator
  'imported'     -- Imported from external dataset
);
```

### 3.3 `features`

The core geospatial table. One row per polygon. Each polygon is a single mapped feature (e.g. "hole 7 green", "hole 3 fairway section 2"). This is where PostGIS earns its keep.

| Column | Type | Constraints | Notes |
|---|---|---|---|
| `id` | UUID | PK, DEFAULT `gen_random_uuid()` | |
| `hole_id` | UUID | NULLABLE, FK → `holes.id` | CASCADE DELETE. NULL = unassigned polygon awaiting reviewer assignment |
| `course_id` | UUID | NOT NULL, FK → `courses.id` | Denormalized for fast course-level queries |
| `feature_type` | `feature_type_enum` | NOT NULL | Enum — see below |
| `geometry` | GEOMETRY(MULTIPOLYGON, 4326) | NOT NULL | PostGIS multipolygon in WGS84 — handles non-contiguous fairways (Decision 1) |
| `area_sqm` | DECIMAL(10,2) | NULLABLE | Precomputed area in m² (via `ST_Area`) |
| `confidence_score` | DECIMAL(4,3) | NULLABLE | Segmentation model confidence |
| `pixel_class_id` | SMALLINT | NULLABLE | Original class ID from DeepLabv3+ output |
| `reviewed` | BOOLEAN | NOT NULL, DEFAULT `false` | Has a human confirmed this polygon? |
| `created_at` | TIMESTAMPTZ | NOT NULL, DEFAULT `NOW()` | |
| `updated_at` | TIMESTAMPTZ | NOT NULL, DEFAULT `NOW()` | |

**Feature Type Enum:**

```sql
CREATE TYPE feature_type_enum AS ENUM (
  'green',
  'fairway',
  'tee_box',
  'bunker',
  'water_hazard'
  -- 'rough' deferred to v2 (Decision 5): absent from Danish training dataset,
  --   model output channels = 6 in v1. Add when v2 model is trained.
  -- 'out_of_bounds' removed: no ML class or pipeline stage produces this value.
  --   PRD 1 defines classes 0–5 with no out_of_bounds class. Add back if a future
  --   data source (e.g. manual import) requires it.
);
```

**Indexes:**

```sql
-- GIST index on geometry — required for fast spatial queries
CREATE INDEX idx_features_geometry
  ON features USING GIST (geometry);

-- Compound index for course-level polygon lookups
CREATE INDEX idx_features_course_type
  ON features (course_id, feature_type);
```

## 4. Entity Relationships

```
organizations
  ├── users    (1 org → N users)
  └── courses  (1 org → N courses)
        └── holes          (1 course → 1–18 holes)
              └── features (1 hole → N polygons, hole_id nullable for unassigned)

courses
  └── pipeline_jobs  (1 course → N job runs)

features
  └── corrections    (1 feature → N correction history entries, feature_id nullable on hard delete)

users
  └── pipeline_jobs  (1 user → N triggered jobs)
  └── corrections    (1 user → N corrections made)
```

## 5. Key Query Patterns

### 5.1 Find feature by GPS point (rangefinder core query)

```sql
SELECT
  h.hole_number,
  f.feature_type,
  ST_Distance(
    f.geometry::geography,
    ST_SetSRID(ST_Point($lng, $lat), 4326)::geography
  ) AS distance_meters
FROM features f
JOIN holes h ON h.id = f.hole_id
WHERE f.course_id = $course_id
  AND ST_Contains(f.geometry, ST_SetSRID(ST_Point($lng, $lat), 4326))
ORDER BY distance_meters ASC
LIMIT 1;
```

### 5.2 Get all polygons for a course (dashboard / review UI)

```sql
SELECT
  f.id,
  f.feature_type,
  h.hole_number,
  f.confidence_score,
  f.reviewed,
  ST_AsGeoJSON(f.geometry) AS geojson
FROM features f
JOIN holes h ON h.id = f.hole_id
WHERE f.course_id = $course_id
ORDER BY h.hole_number, f.feature_type;
```

### 5.3 Get all flagged holes needing review

```sql
SELECT
  c.name AS course_name,
  h.hole_number,
  h.assignment_confidence,
  COUNT(f.id) AS polygon_count
FROM holes h
JOIN courses c ON c.id = h.course_id
JOIN features f ON f.hole_id = h.id
WHERE h.needs_review = true
  AND c.status = 'assigned'
GROUP BY c.name, h.hole_number, h.assignment_confidence
ORDER BY h.assignment_confidence ASC;
```

## 6. Migration Order

> **CRITICAL:** Run migrations in this exact order. Foreign key constraints will fail if tables are created out of sequence.

1. Install PostGIS extension
2. Create all ENUM types
3. Create `organizations` table
4. Create `users` table
5. Create `courses` table
6. Create `holes` table + unique constraint
7. Create `features` table + GIST index
8. Create `pipeline_jobs` table
9. Create `corrections` table
10. Add `updated_at` auto-update triggers on all tables

All open questions from this PRD are resolved.

| # | Question | Decision | Tracker |

| 1 | Single-tenant or multi-tenant? | **Multi-tenant.** `organizations` table added as top-level tenant entity. `org_id` FK on `courses` and `users`. See §3.0, §3.1, §3.6. | Decision 2 |
